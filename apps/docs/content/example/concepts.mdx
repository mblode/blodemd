## Tenants

### What is a tenant

A tenant is a customer, workspace, or organization inside your platform. Every tenant has its own data, configuration, and branding, while sharing the same codebase and deployment.

**Examples**

- Blog platform: each writer runs their own blog
- Documentation platform: each company has its own docs site
- E-commerce platform: each store owner has their own storefront

### Tenant identification strategies

**Subdomain-based**

Extract the tenant from the subdomain (`tenant1.yourapp.com`).

```ts title="middleware.ts"
const hostname = request.headers.get("host");
const subdomain = hostname.split(".")[0]; // "tenant1"
```

**Custom domain-based**

Map a tenant-owned domain (`tenant1.com` → Tenant 1).

```ts title="middleware.ts"
const tenant = await db.tenant.findFirst({
  where: { customDomain: hostname },
});
```

**Path-based**

Use the first URL segment (`/tenant1/dashboard`).

```ts title="middleware.ts"
const pathname = request.nextUrl.pathname;
const tenantSlug = pathname.split("/")[1]; // "tenant1"
```

### Tenant data isolation

- **Database-level**: include tenant IDs in every query.
- **Application-level**: middleware ensures requests only access tenant-scoped data.
- **Edge Config**: cache tenant configuration at the edge for fast lookups.

## Domains

### Wildcard domains

Wildcard domains let you serve every subdomain from a single Vercel project.

- Add `*.yourapp.com` to your project
- Point the apex domain to Vercel nameservers
- Any subdomain routes to your app automatically
- Vercel issues SSL certificates on the fly

### Custom domains

Custom domains let tenants bring their own domain.

- Add `tenant.com` to your project via API
- Tenants configure DNS (CNAME or nameservers)
- Verify ownership with a TXT record when required
- Vercel issues SSL certificates automatically

### SSL certificate issuance

- Wildcard domains: one wildcard certificate covers all subdomains
- Custom domains: one certificate per domain
- Automatic renewal before expiration

### Domain verification

1. Add the domain to your project
2. Vercel provides a unique TXT record
3. Tenant adds the TXT record
4. Verify ownership via SDK or dashboard
5. Certificate issues once verified

## Routing

### How middleware resolves tenants

```ts title="middleware.ts"
export async function middleware(request: NextRequest) {
  const hostname = request.headers.get("host");

  const tenant = await resolveTenant(hostname);

  const response = NextResponse.next();
  response.headers.set("x-tenant-id", tenant.id);

  return response;
}
```

### Request handling flow

1. User visits `tenant1.yourapp.com`
2. Request hits Vercel’s edge network
3. Middleware extracts the subdomain (`tenant1`)
4. Middleware looks up tenant configuration
5. Tenant context is added to request headers
6. Page reads tenant headers and renders tenant-specific data

### Performance considerations

- **Edge Config**: store tenant config at the edge for sub-10ms lookups
- **Caching**: cache tenant lookups in middleware
- **Connection pooling**: reuse DB connections across tenants

## Architecture

### Single deployment serving multiple domains

Multi-tenant architecture means:

- One Next.js codebase
- One Vercel deployment
- Multiple domains (subdomains + custom domains)
- Shared infrastructure and resources
- Tenant-aware routing and data access

### Tenant context

**In middleware**

```ts title="middleware.ts"
response.headers.set("x-tenant-id", tenant.id);
```

**In server components**

```ts title="server-component.ts"
import { headers } from "next/headers";

const tenantId = headers().get("x-tenant-id");
```

**In API routes**

```ts title="api-route.ts"
const tenantId = request.headers.get("x-tenant-id");
```
